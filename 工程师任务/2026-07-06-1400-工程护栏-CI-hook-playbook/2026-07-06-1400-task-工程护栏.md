# 任务：为 grokACP 落地零依赖工程护栏（CI + 本地 pre-commit hook + 护栏 playbook）

## 背景

本项目此前**没有任何 CI**（无 `.github/workflows`），lint/测试全靠人工手跑；整个工作代码库长期未纳入 git（刚刚才补齐 provenance）。这正是 Scott Hanselman 说的“AI 不会破坏健康的工程体系，但会把糟糕的体系照得一清二楚”——护栏缺失时，AI 生成的代码没有强制验证关卡。

本任务把三层护栏一次性补齐，**全部零 npm 依赖**（项目铁律：主仓库不得新增任何 npm 依赖，`ui/` 子目录已有依赖不动）：

1. **CI**（GitHub Actions）：push/PR 时强制跑 lint + 后端测试 + 覆盖率地板 + UI 测试/构建。
2. **本地 pre-commit hook**（native git hook，零依赖，opt-in）：提交前跑 lint + 后端测试。
3. **护栏 playbook**（`docs/guardrails.md`）：把“AI 代码 = 陌生人的 PR”这套 SDLC 纪律固化成项目内文档。

以下所有文件内容**均已由 PM 实测验证**（覆盖率命令实测 exit 0，地板 50/50/65 对应实测 56/59/74），请**逐字创建，不要改动阈值或结构**。

## 只允许修改这些文件

1. `.github/workflows/ci.yml` — **新建**

4. `package.json` — **仅新增一个 `coverage` 脚本**，其余 scripts / 字段一律不动

**禁止**改动其他任何文件。**禁止**新增 npm 依赖。**禁止**改 `src/`、`test/`、`ui/`、`bin/`、`tools/` 下任何代码。**禁止**改动 `package.json` 里除新增 `coverage` 外的任何内容。

## 修改一：`package.json` — 新增 `coverage` 脚本

在 `scripts` 对象里，**紧挨 `"test"` 之后**新增一行 `"coverage"`（其余脚本顺序和内容保持不变）。新增内容**逐字如下**（单引号是命令的一部分，必须保留）：

```json
"coverage": "node --test --experimental-test-coverage --test-coverage-exclude='test/**' --test-coverage-exclude='tools/**' --test-coverage-lines=50 --test-coverage-branches=50 --test-coverage-functions=65 test/**/*.test.mjs",
```

改完后 `scripts` 中 `test` 与 `coverage` 相邻示例（其余脚本略）：

```json
  "test": "node --test test/**/*.test.mjs",
  "coverage": "node --test --experimental-test-coverage --test-coverage-exclude='test/**' --test-coverage-exclude='tools/**' --test-coverage-lines=50 --test-coverage-branches=50 --test-coverage-functions=65 test/**/*.test.mjs",
  "test:watch": "node --test --watch test/**/*.test.mjs",
```

注意：JSON 里最后一个脚本后面不能有多余逗号——`coverage` 后面还有别的脚本，所以带逗号；如果你把它放在末尾则去掉逗号。保持 JSON 合法。

## 修改二：`.github/workflows/ci.yml` — 新建

内容**逐字如下**（可直接采用）：

~~~yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  backend:
    name: lint + test (node ${{ matrix.node }})
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm run lint
      - run: npm test

  coverage:
    name: coverage floor (node 22)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: npm run coverage

  ui:
    name: ui test + build (node 22)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: ui/package-lock.json
      - run: npm ci
        working-directory: ui
      - run: npm run test
        working-directory: ui
      - run: npm run build
        working-directory: ui
~~~

说明（不要写进文件，仅供你理解）：后端零依赖，无需 `npm ci`；覆盖率门禁只在 Node 22 跑（阈值 flag 是 Node 22+ 特性）；UI 用 `npm ci`（`ui/package-lock.json` 已跟踪）。

## 修改四：`docs/guardrails.md` — 新建

内容**逐字如下**（可直接采用；这是一份 markdown，注意整体照抄）：

~~~markdown
# grokACP 工程护栏 Playbook

> 决定软件质量的不是模型参数，而是工程护栏的完整程度。AI 只是把这个事实放大 5–10 倍。
> —— 本项目对 Scott Hanselman “AI reflects your SDLC back at you” 的落地实践。

本文件描述 grokACP 对**所有代码（人类写的、Grok 生成的、子 agent 提交的）一视同仁**的验证纪律。配套的 PM/Grok 协作约束见仓库外 `pm与Grok协作约束文档-v4-ACP优化版.md`。

## 0. 核心原则：把 AI 代码当成“陌生人提交的 PR”

不是不信任，而是建立**可执行的信任机制**。无论代码来源，都必须走完相同的验证循环：lint → 测试 → 覆盖率地板 → 审查 → 来源追溯。

## 1. 三层护栏

| 层 | 机制 | 强制性 | 覆盖 |
|---|---|---|---|
| CI | `.github/workflows/ci.yml`（GitHub Actions） | 强制（push/PR 触发） | lint、后端测试（Node 20+22）、覆盖率地板、UI 测试+构建 |
| 本地 | `.githooks/pre-commit`（native git hook） | opt-in（需手动启用） | lint、后端测试 |
| 流程 | 本 playbook + v4 协作约束 | 人工遵循 | AI 代码审查清单、脱敏、provenance |

## 2. 启用本地 pre-commit hook

零依赖，基于 git 原生 `core.hooksPath`（不引入 husky/lint-staged）：

```bash
git config core.hooksPath .githooks
```

启用后每次 `git commit` 会先跑 lint + 后端测试，失败则拒绝提交。应急跳过：`git commit --no-verify`（仅在明确知道自己在做什么时）。

## 3. CI 强制什么

- **backend**（Node 20 + 22 矩阵）：`npm run lint`（`node --check` 全部 `.mjs`）+ `npm test`（`node:test`，零依赖）。
- **coverage**（Node 22）：`npm run coverage` —— 源码覆盖率地板 `lines≥50 / branches≥50 / funcs≥65`（排除 `test/`、`tools/`）。这是**防倒退地板**，不是目标；正常开发远高于此。
- **ui**（Node 22）：`npm ci` + `vitest run` + `vite build`（含 `tsc -b --noEmit` 类型检查）。

CI 全绿是合入 `main` 的前置条件。

## 4. AI 生成代码合入前清单

把每次 Grok/子 agent 产出当作 PR 过一遍：

- [ ] **范围**：只动了任务白名单内的文件？无未授权 npm 依赖？
- [ ] **验证**：lint 绿、`npm test` 绿、覆盖率未跌破地板？
- [ ] **异常路径**：改了 retry/fallback/超时逻辑时，原 happy path 有回归证明？
- [ ] **Critic 自审**：非小改动是否派了对抗性复审并处理了 CONFIRMED 发现？
- [ ] **敏感数据**：无硬编码密钥；C2C/资金链路原始报告未入库，交付走 `*.redacted.json`（见 v4 硬门禁第 7 条）。
- [ ] **provenance**：改动已 commit、回执/任务文档留痕。

## 5. Provenance（来源追溯）

- **代码必须在 git 里**——未入库的代码不受任何护栏保护。
- 生成物**不入库**：`.codex-artifacts/`（Grok 回执+截图）、`.gitnexus/`（代码图谱索引）、`.hallmark/`、`ui/node_modules`、`ui/dist` 均在 `.gitignore`。
- 任务文档与回执（`工程师任务/`）**入库**，作为工程过程的审计轨迹。

## 6. 零依赖铁律

主仓库不得新增任何 npm 依赖（`dependencies` / `devDependencies` 恒为空）。所有护栏都用 Node 原生能力实现：`node --test`、`node --check`、`--experimental-test-coverage`、native git hook。新增护栏时同样遵守此约束；`ui/` 子目录是唯一例外。
~~~
