## 问题与解决

### 问题 1：UI 黑屏（修正任务）

**症状：** 监控 UI 刷新后黑屏，浏览器 JS bundle 加载失败。

**根因：** `ui/tsconfig.app.json` 的 `include: ["src"]` 包含了 `src/**/*.test.ts`，TypeScript 编译时找不到 `vitest` 类型，导致 `npm run build` 失败。

**解决：** 添加 `"exclude": ["src/**/*.test.ts", "src/**/*.test.tsx"]`。

### 问题 2：npm test 包含 TypeScript 测试文件（修正任务）

**症状：** `npm test` 在根目录运行时尝试编译 `ui/src/lib/format.test.ts`，但 Node.js 原生 `--test` 不支持 TypeScript。

**解决：** 拆分测试脚本：
- `npm test` → `node --test test/**/*.test.mjs`（只跑后端）
- `npm run test:ui` → `npm --prefix ui run test`（只跑前端）
- `npm run test:all` → 同时运行

### 问题 3：T4 首次超时（5 分钟）

**症状：** Task 4 在派发 Critic 子 agent 后超时，回执未生成。

**根因：** Critic 审查两个大文件（`acp-client.mjs` 约 300 行 + `test/acp-client.test.mjs` 约 200 行）需要 5-8 分钟，5 分钟超时不够。

**解决：** 重试，超时改为 20 分钟，成功完成。

## Critic 自审亮点（T4）

Grok 在 T4 完成后自动派发 Critic 子 agent，发现 **11 个真实问题**：

### CONFIRMED 问题（8 个）

**`src/acp-client.mjs`（4 个，未修）：**
1. 进程退出后 `this.proc` 不置空，无法自愈
2. `exit` 处理未关闭 readline
3. `runPrompt` 不重置 `text`/`events`/`stderr`（多次调用会混杂）
4. `waitForStableText` 超时静默返回（不抛错）

**处理决策：** 未修，任务明确"不改实现"，且是设计取舍（CLI 每次 `run` 新建 client，单次 dispatch 模式不受影响）。

**`test/acp-client.test.mjs`（4 个，部分修）：**
1. fake stdin 无 try/catch（非法 JSON 会崩溃） → **已修**
2. fake 绕过 readline/stdout（单元测试局限） → **接受**（端到端 smoke 覆盖真实流）
3. `attachFakeProcess` 重复 `start()` 逻辑 → **接受**（有意为之）
4. fake stdin 不按行缓冲 → **接受**（当前单请求单 write 足够）

### PLAUSIBLE 问题（3 个，2 修 1 不修）

1. 进程退出测试依赖 `setImmediate` 竞态 → **已修**（加 `timeoutMs: 5000` 兜底）
2. `proc.kill` 空操作 → **不修**（测试末尾已 `close()`）
3. 缺 JSON-RPC error / non-JSON stdout 测试 → **不修**（超出范围，可作后续任务）

**Critic 价值：** 发现的问题都是真实风险，Grok 的处理决策合理（区分"实现设计"vs"测试缺陷"）。

## 经验总结

### 成功因素

1. **提示词详细度** — 给 Grok Composer 2.5 的任务文档平均 3-4k tokens，包含完整代码示例、边界约束、验收命令，指令遵循率 100%。
2. **TDD 模式适配** — Composer 2.5 按"先写测试 → 跑红 → 写实现 → 跑绿"流程执行，回执贴两次测试输出，流程清晰。
3. **Critic 自审有效** — T4 的 Critic 发现 11 个真实问题，证明"执行方自审"模式可行。
4. **狗粮模式验证** — 用 grokACP 派发 grokACP 的改进任务，监控 UI 实时显示进度，闭环完整。

### 改进空间

1. **超时设置** — Composer 2.5 处理复杂任务（mock + 测试 + Critic）需要 10-20 分钟，默认 2 分钟太短。建议：
   - 简单任务（纯函数测试）：5 分钟
   - 中等任务（带 mock）：10 分钟
   - 重型任务（含 Critic）：20 分钟
2. **Mock 技术栈** — Node.js 原生 `mock.method()` 无法 mock 内置模块，Grok 最终用"注入 fake process"绕过，可以在 PRD 里预先说明。
3. **测试局限性透明化** — Critic 发现 fake process 绕过 readline，这是单元测试的合理取舍，但应在测试文件注释里说明局限。
